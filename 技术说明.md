# Unlock Music — 完整技术说明文档

## 项目概述

Unlock Music 是一款 QQ 音乐加密歌曲解锁工具。本项目能够解密 QQ 音乐客户端下载的加密歌曲文件（`.mgg`、`.mflac`、`.qmcflac`、`.mmp4` 等），还原为通用无加密音频格式，并通过 QQ 音乐开放 API 自动获取歌曲元数据（歌名、歌手、专辑名称）及专辑封面图片，最终统一转换为带完整 ID3v2.3 标签信息与嵌入式封面的 MP3 音频文件。

## 二、加密算法详解

### 2.1 TEA (Tiny Encryption Algorithm) 完整实现

TEA 是 David Wheeler 和 Roger Needham 于 1994 年提出的轻量级对称分组密码。QQ 音乐使用 TEA 作为核心加密基元。

#### 2.1.1 TeaCipher 类定义

```javascript
class TeaCipher {
  static delta = 0x9E3779B9;      // 黄金分割常数 (2³² / φ)
  static numRounds = 32;          // Feistel 轮数 (标准 TEA 为 64，此处使用半轮)
  
  constructor(key) {
    // 密钥: 16 字节 → 4 个 uint32 (Big-Endian)
    this.k0 = key.getUint32(0, false);  // 字节偏移 0-3
    this.k1 = key.getUint32(4, false);  // 字节偏移 4-7
    this.k2 = key.getUint32(8, false);  // 字节偏移 8-11
    this.k3 = key.getUint32(12, false); // 字节偏移 12-15
    this.rounds = 32;
  }
}
```

**关键参数说明:**

| 参数 | 值 | 数学意义 |
|------|-----|---------|
| `delta` | `0x9E3779B9` | ⌊2³² / φ⌋，黄金分割比衍生常数，用于每轮累加 |
| `numRounds` | `32` | 16 轮全循环 (每轮更新 v0 和 v1 各一次，共 32 次操作) |
| `key length` | `128 bits` | 4 个 uint32 子密钥 |
| `block size` | `64 bits` | 2 个 uint32 (v0, v1) |

#### 2.1.2 解密核心循环

```javascript
decrypt(dst, src) {
  let v0 = src.getUint32(0, false);   // 第一个 32-bit 分组 (Big-Endian)
  let v1 = src.getUint32(4, false);   // 第二个 32-bit 分组 (Big-Endian)
  
  // sum 初始值 = delta × rounds (加密时的最终 sum)
  let sum = (0x9E3779B9 * 32) / 2;
  //       = 0xC6EF3720 / 2
  //       = 0xE3779B90
  
  for (let i = 0; i < 16; i++) {      // 16 轮 (等同于 32 次半轮操作)
    // 解密 v1
    v1 -= ((v0 << 4) + k2) ^ (v0 + sum) ^ ((v0 >>> 5) + k3);
    // 解密 v0
    v0 -= ((v1 << 4) + k0) ^ (v1 + sum) ^ ((v1 >>> 5) + k1);
    // 递减 sum
    sum -= 0x9E3779B9;
  }
  
  dst.setUint32(0, v0, false);
  dst.setUint32(4, v1, false);
}
```

**解密公式分解 (单向):**

```
v1_new = v1 - [ ((v0 << 4) + k₂)  XOR  (v0 + sum)  XOR  ((v0 >> 5) + k₃) ]
         └──────┬──────┘    └───┬───┘    └──────┬──────┘
           线性变换 + 子密钥   加法混淆    非线性移位 + 子密钥

三个分量:
  1. (v0 << 4) + k₂  — 左移 4 位 + 子密钥 2 (扩散)
  2. v0 + sum        — 直接加法混淆 (与轮相关的动态混淆)
  3. (v0 >> 5) + k₃  — 右移 5 位 + 子密钥 3 (非线性压缩)
  
  三项 XOR 组合 → 非线性的轮函数输出
```

#### 2.1.3 CBC 模式的实现

项目中的 TEA CBC 并非标准 CBC，而是自定义的链式 XOR 模式：

```javascript
function decryptTencentTea(inBuf, key) {
  // 前置校验
  if (inBuf.length % 8 != 0) throw Error('inBuf size not multiple of 8');
  if (inBuf.length < 16) throw Error('inBuf size too small');
  
  // 初始化 TEA 密码实例
  const blk = new TeaCipher(key);
  const tmpBuf = new Uint8Array(8);      // 8 字节工作缓冲区
  const tmpView = new DataView(tmpBuf.buffer);
  
  // 解密第一个块获取填充信息
  blk.decrypt(tmpView, new DataView(inBuf.buffer, inBuf.byteOffset, 8));
  const nPadLen = tmpBuf[0] & 0x7;       // 填充长度 (低 3 位, 值 0-7)
  
  // 输出长度 = 总长 - 1(首字节) - nPadLen - 2(SALT) - 7(ZERO)
  const outLen = inBuf.length - 1 - nPadLen - 2 - 7;
  const outBuf = new Uint8Array(outLen);
  
  // CBC 链式变量
  let ivPrev = new Uint8Array(8);        // 前一个密文块 (初始为零)
  let ivCur  = inBuf.slice(0, 8);        // 当前密文块 (初始为块0)
  
  // 处理 SALT 区域 (跳过 2 字节盐值)
  for (let i = 1; i <= 2; ) {
    if (tmpIdx < 8) { tmpIdx++; i++; }
    else cryptBlock();  // 解密下一个块
  }
  
  // 处理 ZERO 与数据区域
  while (outBufPos < outLen) {
    if (tmpIdx < 8) {
      // CBC XOR 解密: 明文 = 解密后的中间值 XOR 前一个密文块
      outBuf[outBufPos] = tmpBuf[tmpIdx] ^ ivPrev[tmpIdx];
      outBufPos++; tmpIdx++;
    }
    else cryptBlock();
  }
}

// cryptBlock: 解密下一个密文块
const cryptBlock = () => {
  ivPrev = ivCur;                              // 当前块变为"前一块"
  ivCur  = inBuf.slice(inBufPos, inBufPos + 8); // 读取新的密文块
  for (let j = 0; j < 8; j++) tmpBuf[j] ^= ivCur[j]; // XOR 链式操作
  blk.decrypt(tmpView, tmpView);              // TEA 解密
  inBufPos += 8;
  tmpIdx = 0;
};
```

**CBC 模式流程:**

```
密文块序列:   C₀    C₁    C₂    C₃    ...
              │     │     │     │
              ▼     ▼     ▼     ▼
           ┌─────┐┌─────┐┌─────┐┌─────┐
           │ TEA ││ TEA ││ TEA ││ TEA │  ← 解密
           └──┬──┘└──┬──┘└──┬──┘└──┬──┘
              │     │     │     │
           ┌──┴──┐┌──┴──┐┌──┴──┐┌──┴──┐
      XOR: │  0  ││ C₀  ││ C₁  ││ C₂  │  ← CBC 链式 XOR
           └──┬──┘└──┬──┘└──┬──┘└──┬──┘
              │     │     │     │
              ▼     ▼     ▼     ▼
             P₀    P₁    P₂    P₃    ...   ← 明文

特殊之处: 第一个块的 "前一个密文块" 初始为零向量 (非随机 IV)
```

**数据格式布局:**

```
字节偏移    │ 内容                    │ 长度
───────────┼────────────────────────┼──────
0           │ 加密的填充长度指示字节   │ 1
1 ~ nPadLen │ 填充字节               │ nPadLen (0-7)
接下来      │ SALT 盐值              │ 2
接下来      │ ZERO 校验区             │ 7
最后        │ 实际音频数据 (明文)      │ outLen
```

### 2.2 EncV2 密钥派生协议

QQ 音乐新版加密格式 (EncV2) 使用双层 TEA 解密 + Base64 解码进行密钥派生。

#### 2.2.1 完整派生流程

```javascript
function decryptV2Key(key) {
  const td = new TextDecoder();
  
  // Step 1: 检测 EncV2 魔数
  if (key.length < 18 || td.decode(key.slice(0, 18)) !== 'QQMusic EncV2,Key:')
    return key;  // 非 EncV2 格式，直接返回原始密钥
  
  // Step 2: 第一层 TEA 解密 (mixKey1)
  let out = decryptTencentTea(key.slice(18), mixKey1);
  
  // Step 3: 第二层 TEA 解密 (mixKey2)  
  out = decryptTencentTea(out, mixKey2);
  
  // Step 4: Base64 解码
  const kd = Buffer.from(td.decode(out), 'base64');
  
  if (kd.length < 16) throw Error('EncV2 key decode failed');
  return kd;
}
```

**五层派生流水线:**

```
原始密钥数据 (18+N 字节)
  │
  ├─ [0:18]   → "QQMusic EncV2,Key:" 魔数校验
  └─ [18:end] → 加密的密钥数据
       │
       ▼
  第一层 TEA 解密 (mixKey1)
       │  密钥: 33 38 36 5A 4A 59 21 40 23 2A 24 25 5E 26 29 28

       │  密钥: 2A 2A 23 21 28 23 24 25 26 5E 61 31 63 5A 2C 54

       │
       ▼

       │
       ▼

```

### 2.3 QMC 密钥派生 (QmcDeriveKey)

```javascript
function QmcDeriveKey(raw) {
  const td = new TextDecoder();
  
  // Step 1: Base64 解码原始密钥
  let rawDec = Buffer.from(td.decode(raw), 'base64');
  if (rawDec.length < 16) throw Error('key length too short');
  
  // Step 2: EncV2 解密 (如果适用)
  rawDec = decryptV2Key(rawDec);
  
  // Step 3: 构建简单密钥 (基于 tan 函数的伪随机序列)
  const simpleKey = [];
  for (let i = 0; i < 8; i++) {
    simpleKey[i] = 0xFF & (Math.abs(Math.tan(106 + i * 0.1)) * 100.0);
  }
  
  // Step 4: 密钥交错混合
  let teaKey = new Uint8Array(16);
  for (let i = 0; i < 8; i++) {
    teaKey[i << 1]       = simpleKey[i];   // 偶数位: 简单密钥
    teaKey[(i << 1) + 1] = rawDec[i];      // 奇数位: 原始密钥
  }
  
  // Step 5: 使用混合密钥解密剩余部分
  const sub = decryptTencentTea(rawDec.subarray(8), teaKey);
  rawDec.set(sub, 8);
  
  return rawDec.subarray(0, 8 + sub.length);  // 返回完整派生密钥
}
```

**简单密钥的数学推导:**

```
simpleKey[i] = floor(|tan(106 + 0.1 × i)| × 100) mod 256

i=0: tan(106.0) × 100 → ...
i=1: tan(106.1) × 100 → ...
...
i=7: tan(106.7) × 100 → ...

共计 8 个字节的固定伪随机序列 (仅依赖角度参数 106 和步长 0.1)
```

### 2.4 QMC Map Cipher (QmcMapCipher)

适用于派生密钥长度 ≤ 300 字节的情况。

#### 2.4.1 核心数据结构

```javascript
class QmcMapCipher {
  constructor(key) {
    this.key = key;   // 密钥字节数组
    this.n   = key.length;
  }
  
  // 8 位循环左移
  static rotate(v, b) {
    let r = (b + 4) % 8;              // 旋转量固定在 [4, 11] 范围内
    return ((v << r) | (v >> r)) & 0xFF; // 循环移位 + 掩码至 8 位
  }
  
  // 获取位置 offset 的掩码字节
  getMask(o) {
    if (o > 0x7FFF) o %= 0x7FFF;     // 偏移大于 32767 时回绕
    const idx = (o * o + 71214) % this.n;  // 二次探测 + 固定偏移
    return QmcMapCipher.rotate(
      this.key[idx],
      idx & 0x7                        // 旋转量取低 3 位
    );
  }
}
```

**散列函数分析:**

```
idx(offset) = (offset² + 71214) mod keyLength

为什么是 71214?
  — 这是一个任意的固定常量，确保 offset=0 时 idx ≠ 0
  — offset² 确保相近位置映射到不同的密钥索引
  — 二次增长使得密钥使用模式高度非线性
```

**rotate 函数分析:**

```
r = (b + 4) % 8  → r ∈ {4, 5, 6, 7, 0, 1, 2, 3}

对于 8 位值 v = b₇b₆b₅b₄b₃b₂b₁b₀:

r=4: b₃b₂b₁b₀b₇b₆b₅b₄  (左移 4 位)
r=5: b₂b₁b₀b₇b₆b₅b₄b₃  (左移 5 位)
r=6: b₁b₀b₇b₆b₅b₄b₃b₂  (左移 6 位)
r=7: b₀b₇b₆b₅b₄b₃b₂b₁  (左移 7 位)
r=0: b₇b₆b₅b₄b₃b₂b₁b₀  (不变)
r=1: b₆b₅b₄b₃b₂b₁b₀b₇  (右移 1 位 = 左移 7 位)
r=2: b₅b₄b₃b₂b₁b₀b₇b₆  (右移 2 位 = 左移 6 位)
r=3: b₄b₃b₂b₁b₀b₇b₆b₅  (右移 3 位 = 左移 5 位)
```

#### 2.4.2 解密过程

```javascript
decrypt(buf, off) {
  for (let i = 0; i < buf.length; i++) {
    buf[i] ^= this.getMask(off + i);  // 逐字节 XOR
  }
}
```

**特点:**
- 每个字节使用位置相关的唯一掩码
- XOR 操作可逆 (加密 = 解密)
- 密钥空间: 256^keyLength (与密钥长度强相关)

### 2.5 QMC RC4 Cipher (QmcRC4Cipher)

适用于派生密钥长度 > 300 字节的情况。

#### 2.5.1 初始化

```javascript
class QmcRC4Cipher {
  static FIRST = 0x80;       // 首段偏移: 128 字节
  static SEG   = 5120;       // 分段大小: 5120 字节
  
  constructor(key) {
    this.key = key;
    this.N   = key.length;   // S-Box 大小 = 密钥长度
    
    // 初始化 S-Box (恒等排列)
    this.S = new Uint8Array(this.N);
    for (let i = 0; i < this.N; i++) this.S[i] = i & 0xFF;
    
    // KSA (Key Scheduling Algorithm) — RC4 密钥调度
    let j = 0;
    for (let i = 0; i < this.N; i++) {
      j = (this.S[i] + j + this.key[i % this.N]) % this.N;
      [this.S[i], this.S[j]] = [this.S[j], this.S[i]];  // 交换
    }
    
    // 计算 hash 因子
    this.hash = 1;
    for (let i = 0; i < this.N; i++) {
      let v = this.key[i];
      if (!v) continue;
      const nh = (this.hash * v) >>> 0;  // 无符号 32 位乘法
      if (nh == 0 || nh <= this.hash) break;  // 溢出或退化时停止
      this.hash = nh;
    }
  }
}
```

**与标准 RC4 的差异:**

| 特性 | 标准 RC4 | QMC RC4 |
|------|---------|---------|
| S-Box 大小 | 固定 256 | 动态 = 密钥长度 |
| KSA 轮数 | 固定 256 | = 密钥长度 |
| 输出方式 | 连续流 | 分段独立状态 |
| 段偏移 | 无 | 每段重新初始化 S-Box |

#### 2.5.2 段密钥计算

```javascript
getSegKey(id) {
  // segKey = floor((hash / ((id+1) × key[id % N])) × 100.0) % N
  return Math.floor(
    (this.hash / ((id + 1) * this.key[id % this.N])) * 100.0
  ) % this.N;
}
```

**公式分解:**

```
对于第 id 个段:
  divisor = (id + 1) × key[id mod N]
  segKey  = floor(hash / divisor × 100) mod N
  
目的: 为每个数据段生成不同的 RC4 初始偏移量
      确保相邻段使用不同的密钥流起始位置
```

#### 2.5.3 分段加密

```javascript
encSeg(buf, off) {
  const S = this.S.slice(0);          // 复制原始 S-Box 状态
  const skip = (off % this.SEG)       // 段内偏移
             + this.getSegKey(Math.floor(off / this.SEG)); // 段密钥偏移
  
  let j = 0, k = 0;
  
  // PRGA with skip: 先跳过 skip 字节，再生成密钥流
  for (let i = -skip; i < buf.length; i++) {
    j = (j + 1) % this.N;
    k = (S[j] + k) % this.N;
    [S[k], S[j]] = [S[j], S[k]];      // S-Box 交换
    if (i >= 0) {
      buf[i] ^= S[(S[j] + S[k]) % this.N];  // 标准 RC4 输出
    }
  }
}
```

**PRGA (Pseudo-Random Generation Algorithm) 分析:**

```
标准 RC4 PRGA:
  i = (i + 1) mod 256
  j = (j + S[i]) mod 256
  swap(S[i], S[j])
  output = S[(S[i] + S[j]) mod 256]

QMC RC4 变体:
  — S-Box 大小变为 N (密钥长度) 而非 256
  — 在生成输出前先 "跳过" skip 字节
  — skip 由段内偏移 + 段密钥偏移组成
  — 其余操作与标准 RC4 一致
```

#### 2.5.4 完整解密流程

```javascript
decrypt(buf, off) {
  let tp = buf.length,           // 剩余待处理字节
      d  = 0;                    // 已处理字节在 buf 中的偏移
  
  const adv = (l) => {
    tp -= l; d += l; off += l;
    return tp == 0;
  };
  
  // 阶段 1: 处理首段 (offset < 128)
  if (off < 0x80) {
    const l = Math.min(buf.length, 0x80 - off);
    for (let i = 0; i < l; i++) {
      buf[i] ^= this.key[this.getSegKey(off + i)];  // 直接 XOR
    }
    if (adv(l)) return;
  }
  
  // 阶段 2: 对齐到段边界
  if (off % 5120 != 0) {
    const l = Math.min(5120 - (off % 5120), tp);
    this.encSeg(buf.subarray(d, d + l), off);
    if (adv(l)) return;
  }
  
  // 阶段 3: 整段处理
  while (tp > 5120) {
    this.encSeg(buf.subarray(d, d + 5120), off);
    adv(5120);
  }
  
  // 阶段 4: 末尾残段处理
  if (tp > 0) {
    this.encSeg(buf.subarray(d), off);
  }
}
```

**三阶段示意:**

```
文件偏移:  0                     0x80               ...           文件尾
           │                     │                               │
           ├──── 阶段1 ──────────┤                               │
           │  直接XOR模式         │                               │
           │  key[segKey(off)]   │                               │
           │                     ├── 阶段2 ──┬── 阶段3 ──────────┤
           │                     │  首段对齐 │  整段 RC4  (×N)   │
           │                     │           │  SEG=5120 字节/段  │
           │                     │           │                    │
           └─────────────────────┴───────────┴────────────────────┘
```

### 2.6 Static Cipher Box (静态回退方案)

当无法从文件尾部提取有效密钥时使用的最终回退方案。

#### 2.6.1 静态 S-Box 定义

```javascript
const STATIC_BOX = new Uint8Array([
  0x77,0x48,0x32,0x73,0xDE,0xF2,0xC0,0xC8,  // 0x00-0x07
  0x95,0xEC,0x30,0xB2,0x51,0xC3,0xE1,0xA0,  // 0x08-0x0F
  0x9E,0xE6,0x9D,0xCF,0xFA,0x7F,0x14,0xD1,  // 0x10-0x17
  // ... (共 256 字节, 索引 0x00-0xFF)
  0x1C,0x71,0xDB,0x00,0xBC,0xFD,0x0C,0x6C,  // 0xF0-0xF7
  0xA5,0x47,0xF7,0xF6,0x00,0x79,0x4A,0x11,  // 0xF8-0xFF
]);
```

#### 2.6.2 解密方式

```javascript
{
  cipher: {
    decrypt(buf, off) {
      for (let i = 0; i < buf.length; i++) {
        let o = off + i;
        if (o > 0x7FFF) o %= 0x7FFF;         // 偏移回绕
        buf[i] ^= STATIC_BOX[(o * o + 27) & 0xFF];  // 二次探测 + 固定偏移 27
      }
    }
  },
  audioSize: fileData.length,   // 假定时整个文件都是音频数据
  songID: 0
}
```

**散列公式:**

```
mask(offset) = STATIC_BOX[(offset² + 27) mod 256]

与 default cipher 的区别:
  — 使用固定 S-Box 而非动态密钥
  — 偏移常量: 27 (default) vs 71214 (map cipher)
  — 无密钥依赖性，安全性最低
```

### 3.2 searchKey() 完整解析逻辑

```javascript
function searchKey(fileData) {
  const last4 = fileData.slice(-4);    // 最后 4 字节
  const td = new TextDecoder();
  const s = td.decode(last4);          // 解码为 ASCII 字符串
  
  // ── 情况 1: 尾标记是 "STag" ──
  if (s === 'STag') {
    throw Error('Android 新版本暂不支持，请降级 QQ 音乐');
  }
  
  // ── 情况 2: 尾标记是 "QTag" ──
  if (s === 'QTag') {
    // 从末尾倒数第 8-4 字节读取密钥块长度 (Big-Endian uint32)
    const ks = new DataView(
      fileData.slice(-8, -4).buffer,
      fileData.slice(-8, -4).byteOffset
    ).getUint32(0, false);
    
    // 音频数据大小 = 文件总长 - 密钥块长度 - 8
    const as = fileData.length - ks - 8;
    
    // 提取密钥块数据
    const rk = fileData.subarray(as, fileData.length - 8);
    
    // 查找第一个 0x2C (逗号) 分隔符
    const ke = rk.findIndex(v => v == 0x2C);
    if (ke < 0) throw Error('密钥格式无效');
    
    // 派生密钥
    const kd = QmcDeriveKey(rk.subarray(0, ke));
    
    // 选择 Cipher (根据密钥长度自适应)
    const c = kd.length > 300
      ? new QmcRC4Cipher(kd)
      : new QmcMapCipher(kd);
    
    // 提取 songID (在两个 0x2C 之间)
    const ib = rk.subarray(ke + 1);                      // 跳过分隔符
    const ie = ib.findIndex(v => v == 0x2C);              // 查找第二个分隔符
    const songID = ie >= 0
      ? parseInt(td.decode(ib.subarray(0, ie)), 10)
      : 0;
    
    return { cipher: c, audioSize: as, songID };
  }
  
  // ── 情况 3: 旧版格式 (尾标记为 Little-Endian uint32) ──
  const ks = new DataView(last4.buffer, last4.byteOffset).getUint32(0, true);
  
  if (ks < 0x400) {   // 密钥块长度 < 1024 (合理性校验)
    const as = fileData.length - ks - 4;
    const rk = fileData.subarray(as, fileData.length - 4);
    const kd = QmcDeriveKey(rk);
    const c = kd.length > 300
      ? new QmcRC4Cipher(kd)
      : new QmcMapCipher(kd);
    return { cipher: c, audioSize: as, songID: 0 };
  }
  
  // ── 情况 4: 最终回退 (Static Cipher Box) ──
  return {
    cipher: {
      decrypt(buf, off) {
        for (let i = 0; i < buf.length; i++) {
          let o = off + i;
          if (o > 0x7FFF) o %= 0x7FFF;
          buf[i] ^= STATIC_BOX[(o * o + 27) & 0xFF];
        }
      }
    },
    audioSize: fileData.length,
    songID: 0
  };
}
```

**决策流程图:**

```
文件最后 4 字节解码
  │
  ├─ == "STag" ───→ ❌ 抛出异常: "Android 新版本暂不支持"
  │
  ├─ == "QTag" ───→ 读取 Big-Endian ks
  │                   │
  │                   ├─ 密钥长度 > 300 ──→ QmcRC4Cipher
  │                   └─ 密钥长度 ≤ 300 ──→ QmcMapCipher
  │
  ├─ 解析为 uint32 LE → ks < 0x400?
  │   │
  │   ├─ 是 ──→ 旧版 QMC: 派生密钥 → RC4/Map Cipher
  │   │
  │   └─ 否 ──→ Static Box 回退

```

```javascript
function getDuration(inputPath, ffmpegPath) {
  const ffprobe = ffmpegPath.replace(/ffmpeg(\.exe)?$/, 'ffprobe$1');
  
  // 方法 1: ffprobe 直接提取 duration 元数据
  try {
    const out = execFileSync(ffprobe, [
      '-v', 'error',                    // 仅输出错误
      '-show_entries', 'format=duration', // 仅提取时长字段
      '-of', 'csv=p=0',                // CSV 格式, 无表头
      inputPath
    ], { stdio: 'pipe', timeout: 15000 });
    const d = parseFloat(out.toString().trim());
    if (d > 0) return d;
  } catch {}
  
  // 方法 2: ffmpeg 解码输出中解析 Duration 行
  try {
    execFileSync(ffmpegPath, ['-i', inputPath], {
      stdio: 'pipe',
      timeout: 15000
    });
  } catch (e) {
    // ffmpeg -i 无输出文件时会以非零码退出，但 stderr 中包含文件信息
    const output = (e.stderr || e.stdout || '').toString();
    const m = output.match(/Duration:\s*(\d+):(\d+):(\d+)\.(\d+)/);
    if (m) {
      return parseInt(m[1]) * 3600   // 时 → 秒
           + parseInt(m[2]) * 60     // 分 → 秒
           + parseInt(m[3])          // 秒
           + parseInt(m[4]) / 100;   // 百分之一秒 → 秒
    }
  }
  
  return 0;  // 无法检测
}
```

正则捕获组:

```javascript
const report = (sec) => {
  const pct = totalSec > 0 ? Math.min(sec / totalSec, 1) : 0;
  const pct1 = Math.round(pct * 1000) / 10;  // 保留 1 位小数
  
  // 节流: 百分比未变化时不更新
  if (pct1 === lastPct && pct1 < 100) return;
  lastPct = pct1;
  
  // 进度条填充
  const filled = Math.round(barW * pct);
  const bar = GREEN + '#'.repeat(filled) + RESET
            + DIM + '-'.repeat(barW - filled) + RESET;
  
  // 百分比字符串 (6 字符右对齐)
  const pctStr = BOLD + String(pct1.toFixed(1)).padStart(6) + '%' + RESET;
  
  // ETA 计算
  let eta = '--';
  if (totalSec > 0 && sec > 0) {
    eta = Math.max(0,
      (totalSec - sec) / sec * (Date.now() - startTime) / 1000
    ).toFixed(0) + 's';
  }
  
  onProgress(bar + ' ' + pctStr + DIM + ' 剩' + eta + RESET);
};
```

**ETA 公式推导:**

```
如果:
  — totalSec = 总时长 (秒)
  — sec = 已处理时长 (秒)
  — elapsed = 已耗时 (秒)

剩余比例 = (totalSec - sec) / sec
ETA = 剩余比例 × elapsed = (totalSec - sec) / sec × elapsed

示例:
  totalSec = 263.58s
  sec = 100s
  elapsed = 30s
  
  剩余比例 = (263.58 - 100) / 100 = 1.6358
  ETA = 1.6358 × 30 = 49.074s → 显示 "49s"
```

### 5.5 HTTP 请求实现

```javascript
function httpGet(url, timeout = 10000) {
  return new Promise((resolve, reject) => {
    const mod = url.startsWith('https') ? https : http;
    
    const req = mod.get(url, { timeout }, (res) => {
      // 自动跟随重定向 (301/302/303/307/308)
      if (res.statusCode >= 300 && res.statusCode < 400 && res.headers.location) {
        return resolve(httpGet(res.headers.location, timeout));
      }
      
      // 非成功状态码
      if (res.statusCode < 200 || res.statusCode >= 300) {
        req.destroy();
        return reject(new Error('HTTP ' + res.statusCode));
      }
      
      // 收集响应体
      const chunks = [];
      res.on('data', c => chunks.push(c));
      res.on('end', () => resolve({
        status:  res.statusCode,
        data:    Buffer.concat(chunks),
        headers: res.headers
      }));
    });
    
    req.on('error', reject);
    req.on('timeout', () => {
      req.destroy();
      reject(new Error('timeout'));
    });
  });
}
```

### 5.6 文件名元数据解析

```javascript
function parseFilenameMeta(filename) {
  const name = path.basename(filename, path.extname(filename));
  // "G_E_M_ 邓紫棋 - 句号.mgg" → "G_E_M_ 邓紫棋 - 句号"
  
  // 匹配三种分隔符: " - ", " – ", "—"
  const m = name.match(/^(.+?)\s*[-–—]\s*(.+)$/);
  if (!m) return null;
  
  let artist = m[1].trim();
  let title  = m[2].trim();
  
  // 规范化歌手名称: 下划线 → 空格
  artist = artist.replace(/_/g, ' ').replace(/\s+/g, ' ').trim();
  title  = title.replace(/\s+/g, ' ').trim();
  
  if (!artist || !title) return null;
  
  return { artist, title, album: '', albumMid: '', coverUrl: '' };
}
```

**指数缓动曲线详细分析:**

```
pct(t) = 1 - e^{-k·t}    其中 k = 2.2

微分分析:
  dpct/dt = k · e^{-k·t} = k · (1 - pct)
  
  t=0.0s:  pct=0.000,  速度=2.200/s  (最快)
  t=0.2s:  pct=0.356,  速度=1.417/s
  t=0.5s:  pct=0.667,  速度=0.732/s
  t=1.0s:  pct=0.889,  速度=0.244/s
  t=1.5s:  pct=0.963→0.92,  速度=0.082/s  (已达上限)
  t=2.0s:  pct=0.92 (上限), 速度=0/s       (保持上限)

为什么上限设在 92%?
  — 留出视觉空间让用户知道任务还在进行中
  — 100% 是真正的 "完成" 信号，只在实际完成时显示
  — 避免用户在 99% 时以为卡住了
```

**为什么是 k=2.2?**

```
k 值选择权衡:
  — k 太小 (如 0.5): 动画过慢，用户以为卡住
  — k 太大 (如 5.0): 瞬间到 92%，动画无意义
  — k=2.2: 约 1.5 秒到达 92% 上限，给 API 请求留出合理时间窗口

到达上限时间:
  1 - e^{-2.2t} = 0.92
  e^{-2.2t} = 0.08
  -2.2t = ln(0.08)
  -2.2t = -2.526
  t = 1.148 秒

所以动画在约 1.15 秒后到达 92% 上限，之后保持直到实际完成。
```

### 8.4 视觉宽度计算 (CJK 字符适配)

```javascript
function visualWidth(s) {
  // 先剥离 ANSI 转义序列, 只计算可见字符
  const clean = s.replace(/\x1b\[[0-9;]*m/g, '');
  
  let w = 0;
  for (const ch of clean) {
    const c = ch.codePointAt(0);
    
    // CJK 字符占 2 列宽度
    if (
      (c >= 0x1100  && c <= 0x115F)  ||  // Hangul Jamo
      (c >= 0x2E80  && c <= 0xA4CF)  ||  // CJK Radicals ~ Yi
      (c >= 0xAC00  && c <= 0xD7A3)  ||  // Hangul Syllables
      (c >= 0xF900  && c <= 0xFAFF)  ||  // CJK Compatibility
      (c >= 0xFF01  && c <= 0xFF60)  ||  // Fullwidth Forms
      (c >= 0xFFE0  && c <= 0xFFE6)  ||  // Fullwidth Signs
      (c >= 0x20000 && c <= 0x2FFFD) ||  // CJK Ext B ~
      (c >= 0x30000 && c <= 0x3FFFD) ||  // CJK Ext G ~
      (c >= 0x1F000 && c <= 0x1FFFF)     // Emoji / Symbols
    ) {
      w += 2;
    } else {
      w += 1;
    }
  }
  return w;
}
```

**视觉填充函数 (vpad):**

**Batch ESC 获取原理解析:**

## 九、错误处理与容错体系

### 9.1 分层错误处理

```
错误层级:

L1: 致命错误 (进程退出)
  ├── STag 格式不支持 → 明确报错 + 提示降级客户端
  ├── 密钥格式无效   → 提示文件可能损坏
  └── EncV2 解码失败  → 提示密钥解析失败

L2: 文件级错误 (记录失败, 继续处理下一个)
  ├── 加密文件格式无法识别 → 回退 Static Box
  ├── FFmpeg 转码异常       → 标记 ERR, 跳过该文件
  └── API 请求超时/异常     → 降级到下一级元数据策略

L3: 元数据级错误 (丢失元数据, 音频文件不受影响)
  ├── songID API 无响应      → 回退到文件名搜索
  ├── 文件名搜索无结果        → 回退到纯文件名解析
  └── 封面下载失败            → 无封面嵌入, 音频正常

L4: 环境级警告 (仅提示, 功能降级)
  ├── FFmpeg 未找到           → 自动切换 --no-mp3 模式
  ├── 无匹配加密文件          → 显示帮助信息后退出
  └── 网络不可用              → --no-meta --no-cover 行为
```

### 9.2 具体错误场景处理

| 场景 | 异常类型 | 处理方式 |
|------|---------|---------|
| 文件尾部为 "STag" | `throw Error` | 进程终止, 显示降级提示 |
| 密钥块中无 0x2C 分隔符 | `throw Error` | 进程终止, "密钥格式无效" |
| EncV2 Base64 解码失败 | `throw Error` | 进程终止, "EncV2 key decode failed" |
| TEA 解密输入长度非 8 的倍数 | `throw Error` | 进程终止 |
| TEA 解密输入长度 < 16 | `throw Error` | 进程终止 |
| API 返回非 200 状态码 | `reject` → `catch` | 静默回退到下一级策略 |
| API 返回空列表 | `return null` | 静默回退 |
| HTTP 请求超时 | `reject` → `catch` | 静默回退 |
| FFmpeg 转码非零退出 | `reject` → `catch` | 标记为失败, 继续下一个 |
| 封面下载失败 | `resolve(false)` | 无封面嵌入, 音频正常 |
| 系统无可用的网络接口 | `reject` → `catch` | 无元数据/封面, 音频正常 |

### 9.3 用户友好的错误提示

| 内部错误 | 用户可见提示 |
|---------|------------|
| `'Android 新版本暂不支持'` | 完整显示, 附降级指引 |
| `'密钥格式无效'` | "密钥格式无效" + 建议重新下载 |

## 十、性能优化策略

### 10.1 I/O 优化

| 策略 | 实现 | 收益 |
|------|------|------|
| 内存中解密 | `fs.readFileSync()` 一次性加载整个文件到 Buffer, 原地解密 | 避免临时文件 I/O |
| 流式转码输出 | FFmpeg 直接输出到最终 MP3 路径 | 减少一次文件复制 |
| 流复制嵌入元数据 | `-c copy` 而非重编码 | 嵌入元数据耗时从分钟级降到秒级 |
| 临时文件及时清理 | 解密中间文件转码后立即 `fs.unlinkSync()` | 节省磁盘空间 |

### 10.2 并发优化

| 策略 | 实现 | 收益 |
|------|------|------|
| 固定并发池 | 5 个 Worker + `Promise.all()` | 平衡 CPU / I/O / 内存 |
| 共享计数器竞争任务 | `nextIdx++` 原子操作 (单线程 JS 自然原子) | 负载均衡, 无需任务队列 |
| 两个独立并发阶段 | Phase 1 → await → Phase 2 | 先确保音频就绪, 再处理元数据 |

### 10.3 终端 I/O 优化

| 策略 | 实现 |
|------|------|
| 进度更新节流 | 百分比值变化才调用 `writeInline` |
| 光标跳跃而非全屏刷新 | `\x1b[nA` / `\x1b[nB` 定点更新指定行 |
| setInterval 80ms 最小间隔 | 避免 60fps 级别的过度刷新 |

### 10.4 网络优化

| 策略 | 实现 |
|------|------|
| 连接复用 | 使用 `http.globalAgent` / `https.globalAgent` (Node.js 默认 keep-alive) |
| 超时控制 | API 10s / 封面 15s 超时, 防止无限等待 |
| 重定向跟随 | 递归调用 `httpGet(res.headers.location)`, 最多由 Node.js 默认的 10 次限制 |
